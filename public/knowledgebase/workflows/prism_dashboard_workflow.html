  1. Trigger detection — match user phrase to mode (triage, morning-brief, weekly-review, single-ticket). Different modes change report header and scope, not the underlying loop.
  2. Queue pull (parallel) — GET Prism Dashboard API endpoints + Notion task DB + meeting-note action items. Probe each source on first call of a session.
  3. Source-failure branch — if either source is unreachable, log it in the report's "Sources" footer and continue with the working source. Never block.
  4. De-dupe + normalize — merge tickets matching on title+assignee or explicit cross-reference. Output schema: {id, source, title, body, owner, due, age_days, links}.
  5. Open filter — drop dashboard status in {done, closed, cancelled, archived} and Notion status in {Done, Cancelled, Archived}. Overdue still counts as open.
  6. Triage waterfall — 7-rule keyword cascade, first match wins (client-delivery → sales-bd → finance-billing → marketing → research-analytics → engineering → internal-admin catch-all). Cross-functional tickets assigned by primary owner; 3+ specialty spans split into sub-tickets.
  7. Dispatch — parallelize tickets on independent systems, serialize on same record. Each specialist loads its own SKILL.md playbook.
  8. Hard-stop check — any external send, money movement, third-party calendar invite, hiring/comp, new vendor or vendor spend >$500 → escalate without executing.
  9. Two-question test — (a) Is the right action obvious? (b) Is the downside reversible in <5 min? Both yes → execute. Either no → escalate.
  10. Execution — specialist runs action, captures artifact path + confidence (high/med/low).
  11. Size branch — if >30 tickets in sweep, write full report to Development/dashboard/reports/cos-{YYYY-MM-DD}.md and inline only the exec summary + all escalations + top 10 done.
  12. Final report — escalations always first, then Done grouped by specialist, then Not-yet-handled, then Sources footer with endpoints used and any unreachable.

Handoffs between specialists

sales → client-delivery if prospect already a client.
admin (receipt) → finance for journal entry.
research → client-delivery when analysis is a deliverable.

Parallel vs. serial dispatch

COS runs tickets in parallel when they touch independent systems, and serializes tickets that touch the same record (e.g., two updates to the same deal).
