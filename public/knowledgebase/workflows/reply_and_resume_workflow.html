Endpoint

POST /api/tickets/:id/comments — defined at server.js:4896-5000. Accepts { comment, author?, status? }. Single endpoint handles both the agent-side tag flow (D2) and Michele's reply flow (D1).

The atomicity guarantee

SQLite path runs inside db.transaction() — comment + status flip roll back together. Notion path can't be truly atomic across two API calls; runs comment first, then status (best-effort, caller can retry).

D1 wins over D2

If a reply happens to contain [NEEDS-DECISION] AND an explicit status, the explicit status wins. Michele's reply can't be clobbered by a stray tag in her own comment body.

Two surface integrations

Review-status tickets count up the Needs-Decision tile on the Daily Agenda (tiles.needs_decision) and add a red badge to the Tickets nav. Reply & Resume composer renders only when ticket.status === 'review'.

The two write paths are NOT semantically equal. SQLite path is fully atomic — if the status flip fails, the comment is rolled back. Notion path posts the comment first and then attempts the status flip; if the status call fails, the comment is already in Notion and the caller sees the error. Reply & Resume frontend handler reports failures via toast() and re-enables the submit button so Michele can retry.

  1. Validator gate — express-validator rejects requests missing comment (or violating any registered constraints) with 400 + { ok: false, errors }. The handler never proceeds past this on bad input.
  2. Author resolution — req.user?.username (from the Bearer-token auth middleware) wins; falls back to req.body.author, then 'system'. The composer leaves author unset, so authenticated requests carry the logged-in user automatically.
  3. Phase D1 — explicit status path — if the request includes a status field, statusPatch takes that value and statusReason = 'explicit'. Reply & Resume always lands here with status: 'in_progress'. This branch wins over D2 even when both conditions are true.
  4. Phase D2 — tag auto-flip — only checked when D1 didn't fire. If the comment body contains the literal token [NEEDS-DECISION] (case-sensitive, .includes()), statusPatch = 'review' and statusReason = 'needs-decision-tag'. This is how agents move a ticket to Review without a separate PATCH request.
  5. No-op branch — comments without an explicit status and without the tag leave statusPatch undefined; only the comment is written, status is untouched.
  6. SQLite write (local backend) — the entire write is wrapped in db.transaction(): insert into ticket_comments, then UPDATE tickets if statusPatch is set. Either both land or neither does.
  7. Notion write (source-of-truth backend) — Notion has no transactional API across two operations. The handler posts the comment first, then attempts the status PATCH. If the status PATCH fails, the comment is already in Notion; the caller sees a 500 and can retry the status flip explicitly.
  8. Response shape — success returns 201 { ok: true, comment, status, reason }. reason is the canary that tells the client which branch fired ('explicit', 'needs-decision-tag', or absent for no-op).
  9. Frontend cleanup — ticketReplyAndResume() re-fetches the ticket on success, re-renders the modal (which now shows In Progress and the new comment in the thread), and toasts the failure path with toast('Reply failed: ' + e.message, true). The submit button is restored to its default label in a finally block so retries are always available.

The protocol this implements

Workspace-root CLAUDE.md "Ticket discipline" section: "When you need a decision mid-work, POST the specific question to /api/tickets/:id/comments with the literal token [NEEDS-DECISION] somewhere in the comment body. The server auto-flips Status to Review in the same request — no separate PATCH needed."

Why D1 wins over D2

Without this rule, Michele's reply could contain [NEEDS-DECISION] (quoting the agent's question, or referencing the convention), and the server would flip the ticket back to Review immediately — clobbering her own resume action. D1's explicit status always wins.
